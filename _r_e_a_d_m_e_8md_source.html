9 [Causal Dynamical Triangulations][1] in C++ using the
10 [Computational Geometry Algorithms Library][cgal] and [Eigen][eigen]>3.1.0, compiled
11 with [CMake][cmake] using [Clang][clang]/[LLVM][llvm].
12 Arbitrary-precision numbers and functions by [MPFR][mpfr] and [GMP][gmp].
13 Option-types via [Boost][boost] >= 1.63.0.
14 [Docopt][docopt] provides a beautiful command-line interface.
15 [Gmock 1.8][gmock] may be optionally installed in order to build/run unit tests.
16 [Ninja][ninja] is a nice (but optional) replacement for `make`.
17 Intel's [TBB][tbb] provides significantly better performance if present (3x+).
18 Follows the [CppCore Guidelines][guidelines] as enforced by [clang-tidy][tidy], which you can run using the
19 [clang-tidy.sh][8] script:
20 
21 ~~~
22 ./clang-tidy.sh
23 ~~~

56 ### Prerequisites ###
57 ------
58 
59 [CDT++][38] should build on any system (e.g. Linux, MacOS, Windows) with
60 [CMake][14], [CGAL][cgal], [Boost][boost], [MPFR][mpfr], and [Eigen][eigen] installed.
61 [TBB][tbb] provides an optional (but significant) speed boost. [Gmock][gmock] is
62 optional (but recommended) for running the unit tests, and [Ninja][ninja] is an
63 optional replacement for `make` which provides quick parallel builds of the
64 unit tests.
65 
66 On MacOS, the easiest way to do this is with [HomeBrew][homebrew]:
67 
68 ~~~
69 brew upgrade cmake
70 brew install ninja
71 brew upgrade boost
72 brew install eigen
73 brew install tbb --c++11
74 brew install cgal --imaging --with-eigen3 --with-lapack
75 ~~~
76 
77 On Ubuntu, you will need an updated versions of [clang][clang] or [gcc][gcc],
78 [CMake][cmake], and [Boost][boost], which is scripted in [.travis.yml][39].

80 ### Build ###
81 ------
82 This project uses a separate `build/` directory, which allows you to rebuild the
83 project without cluttering the source code. Thus, download this source code
84 (clone this repo from GitHub or grab a zipped archive [here][17]) and run the
85 following commands in the top-level directory:
86 
87 ~~~
88 mkdir build && cd build
89 cmake -DCMAKE_BUILD_TYPE=Release ..
90 cmake --build .
91 ~~~
92 
93 (Or run [build.sh][27] if you have [Ninja][ninja] installed.)
94 
95 This should result in the main program executable, `cdt` in the `build/`
96 directory, along with several others.
97 
98 * `cdt-gv` converts output files to GeomView format for visualization
99 * `cdt-opt` is a simplified version with hard-coded inputs, mainly useful for
100 debugging and scripting
101 
102 If you have [GMock][gmock] installed and build unit tests, the executable
103 `unittests` will also be present. See [Tests](#tests) for details.
104 
105 For some versions of Linux, you may have to build [CGAL][cgal] from source.
106 Follow the instructions (or their equivalent) given in the install section
107 of the [.travis.yml][39] build file.
108 
109 There are enough unit tests that it's worthwhile doing fast parallel builds.
110 [Ninja][ninja] is just the ticket. It's effectively a drop-in replacement for
111 `make`, and works nicely because [CMake][cmake] generates the build files.
112 There's quite a difference in speed. CMake also [abstracts away the build][47]
113 tool nicely. Again, see [build.sh][27] for an example.
114 
115 Possible build troubles:
116 -----------------------
117 * While running `build.sh` under linux with `gcc` you may encounter
118 `virtual memory exhausted: Cannot allocate memory`, a common solution for
119 which is to
120 [allocate swap space](http://www.cyberciti.biz/faq/linux-add-a-swap-file-howto/).

122 ### Usage ###
123 ------
124 CDT-plusplus uses [docopt][docopt] to parse options from the help message, and so
125 understands long or short argument formats, provided the short argument given
126 is an unambiguous match to a longer one. The help message should be instructive:
127 
128 ~~~
129 ./build/cdt --help
130 Causal Dynamical Triangulations in C++ using CGAL.
131 
132 Copyright (c) 2014-2017 Adam Getchell
133 
134 A program that generates d-dimensional triangulated spacetimes
135 with a defined causal structure and evolves them according
136 to the Metropolis algorithm. Specify the number of passes to control
137 how much evolution is desired. Each pass attempts a number of ergodic
138 moves equal to the number of simplices in the simulation.
139 
140 Usage:./cdt (--spherical | --toroidal) -n SIMPLICES -t TIMESLICES [-d DIM] -k K --alpha ALPHA --lambda LAMBDA [-p PASSES] [-c CHECKPOINT]
141 
142 Examples:
143 ./cdt --spherical -n 64000 -t 256 --alpha 0.6 -k 1.1 --lambda 0.1 --passes 1000
144 ./cdt --s -n64000 -t256 -a.6 -k1.1 -l.1 -p1000
145 
146 Options:
147  -h --help Show this message
148  --version Show program version
149  -n SIMPLICES Approximate number of simplices
150  -t TIMESLICES Number of timeslices
151  -d DIM Dimensionality [default: 3]
152  -a --alpha ALPHA Negative squared geodesic length of 1-d
153  timelike edges
154  -k K K = 1/(8*pi*G_newton)
155  -l --lambda LAMBDA K * Cosmological constant
156  -p --passes PASSES Number of passes [default: 100]
157  -c --checkpoint CHECKPOINT Checkpoint every n passes [default: 10]
158 ~~~
159 
160 The dimensionality of the spacetime is such that each slice of spacetime is
161 `d-1`-dimensional, so setting `d=3` generates 2 spacelike dimensions and one
162 timelike dimension, with a defined global time foliation. Thus a
163 `d`-dimensional simplex will have some `d-1` sub-simplices that are purely
164 spacelike (all on the same timeslice) as well as some that are timelike
165 (span two timeslices). In [CDT][1] we actually care more about the timelike
166 links (in 2+1 spacetime) and the timelike faces (in 3+1 spacetime).

188 ### Tests ###
189 -----------
190 [GMock][gmock] is optional, but strongly recommended if you want to make changes to
191 or understand the source code in detail. Building the [GMock][gmock] `unittests`
192 executable is set by the `GMOCK_TESTS` variable in [CMakeLists.txt][28], or
193 at the command line using:
194 
195 ~~~
196 cmake -DGMOCK_TESTS:BOOL=ON -DCMAKE_BUILD_TYPE=Debug ..
197 ~~~
198 
199 To install GMock, you'll need to install GMock and GTest as a shared library.
200 First look at the [README][24] to understand the general idea. On Linux:
201 
202 ~~~
203 git clone https://github.com/google/googletest.git
204 cd googletest/googlemock
205 cmake -DBUILD_SHARED_LIBS=ON .
206 make
207 sudo cp -a include/gmock/ /usr/include/
208 sudo cp -a libgmock_main.so libgmock.so /usr/lib/
209 ~~~
210 
211 GoogleMock is modular, and can use other testing frameworks. Since we want
212 to use GoogleTest, do:
213 
214 ~~~
215 cd ../googletest
216 cmake -DBUILD_SHARED_LIBS=ON .
217 make
218 sudo cp -a include/gtest /usr/include/
219 sudo cp -a libgtest_main.so libgtest.so /usr/lib/
220 ~~~
221 
222 This is scripted in [.travis.yml][39]. (Thanks to [ManuelSchneid3r][42] and
223 [Stack Overflow][40] for the assist.)
224 
225 Unit tests using GMock are then run (in the `build/` directory) via:
226 
227 ~~~
228 ./unittests
229 ~~~
230 
231 You can also run function tests (using [CTest][ctest]) via:
232 
233 ~~~
234 cmake --build . --target test
235 ~~~
236 
237 In addition to the command line output, you can see detailed results in the
238 `build/Testing` directory which is generated thereby.

240 ### Static Analysis ###
241 -----------
242 The [cppcheck-build.sh][35] script runs a quick static analysis using
243 [cppcheck][cppcheck].
244 
245 [Clang][clang] comes with [scan-build][32] which can run a much more thorough,
246 but slower static analysis integrated with [CMake][cmake] and [Ninja][ninja].
247 Simply run the [scan.sh][33] script.
248 
249 Also, these tools build in **DEBUG** mode. You should probably not then run
250 `unittests` as you will get thousands of lines of debugging output from the
251 tests that create large triangulations. (You could use `--gtest_filter`
252 to run just the tests that you want.)
253 
254 One of the [Travis-CI][travis] jobs runs [Valgrind][valgrind]; be sure to look at the
255 results to ensure you're not leaking memory.
